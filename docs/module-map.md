# 模块地图

> 本文档是 AI coding agent 和工程师进入代码库前的系统导航入口。它帮助判断业务变更涉及哪些模块、哪些边界不能破、下一步该读哪些文档或搜索哪些代码。
> 它不是文件树、README、ADR 或完整架构说明。

## 使用方式

- 先根据入口判断改动属于 Web 管理端、员工自助端、API Route、后台任务，还是外部系统集成。
- 涉及新功能、破坏性变更、架构调整、重大性能或安全工作时，先读对应提案、规约或任务文档。
- 具体实现细节继续使用语义搜索、`rg/grep`、模块子文档和代码本身确认。
- 涉及员工生命周期、组织、考勤、薪酬、绩效、权限或通知时，继续读取 `docs/impact-map.md`。

## 关键支撑域

| 模块               | 路径                                                    | 职责                                         | 下一步                               |
| ------------------ | ------------------------------------------------------- | -------------------------------------------- | ------------------------------------ |
| Web UI             | `src/app/`, `src/components/`, `src/features/`          | 承载 Next.js 页面、React 组件和前端交互      | 先确认页面所属业务模块               |
| API Route          | `src/app/api/**/route.ts`                               | 承载 HTTP 入口、认证授权、参数校验和响应转换 | 复杂业务下沉到对应模块 service       |
| Schema 与 DTO      | `src/server/schemas/`, `src/modules/**/dto.ts`          | 定义输入校验、输出形态和接口数据转换         | 与 Route Handler 和 service 配套使用 |
| Service / Use Case | `src/modules/**/services/`, `src/modules/**/use-cases/` | 承载业务编排、事务边界和跨 repository 协作   | 多表写入或跨模块流程优先进入此层     |
| Repository / DB    | `src/server/db/`, `src/modules/**/repositories/`        | 封装数据访问、查询条件和持久化细节           | 不承载业务流程判断                   |
| Workflow           | `src/modules/workflow/`                                 | 承载审批流、任务流转和待办状态               | 涉及审批链时检查组织和权限影响       |
| Auth               | `src/modules/auth/`                                     | 管理登录身份、角色、数据范围和审计上下文     | 涉及敏感数据时先确认权限边界         |
| Notification       | `src/modules/notification/`                             | 管理邮件、站内信、IM 等通知模板和发送记录    | 涉及异步通知时检查幂等和补发         |

## 跨模块硬边界

- React 组件和页面不得直接访问数据库或 repository；需要通过 API client、server action 或明确的业务 service。
- `route.ts` 只处理 HTTP 层职责，不承载复杂业务编排、事务流程或跨模块状态修改。
- Service / Use Case 可以协调多个 repository，但不得返回 `NextResponse`、`Response` 或 UI 专用结构。
- Repository 只封装数据访问，不判断入职、离职、计薪、绩效等业务流程。
- 薪酬模块的金额、计薪周期和工资单锁定规则不得被考勤、绩效或员工模块直接改写；需要通过 Payroll service 或明确事件契约。
- 员工生命周期变更必须同时检查账号、组织、考勤、薪酬、绩效和通知影响。
- 敏感字段如薪资、证件、绩效结果不得在未确认权限和数据范围前返回给前端。

## 高风险影响摘要

> 本节只保留少数最高风险的跨模块影响提示。完整变更影响关系见 `docs/impact-map.md`。

### 候选人转员工

- **如果修改**: 入职审批、员工编号、候选人字段复制、入职撤回
- **先检查**: 员工主档、账号、考勤档案、薪资档案、通知
- **原因**: 入职会一次性创建多个下游档案，遗漏会造成员工状态不完整。

### 组织和岗位生效日期

- **如果修改**: 部门、岗位、汇报关系、成本中心、生效日期
- **先检查**: 考勤审批链、薪资成本归属、绩效评审关系、数据权限
- **原因**: 多个模块按不同日期口径读取组织快照，容易产生历史归属错误。

### 考勤汇总

- **如果修改**: 排班、假期、加班、月度考勤汇总
- **先检查**: 薪资扣款与加班费、员工自助端、考勤报表
- **原因**: 考勤结果常作为薪资输入，修正或锁定时机不一致会导致工资单错误。

### 薪酬金额和周期

- **如果修改**: 薪资项、金额精度、计薪周期、工资单锁定
- **先检查**: 工资单、财务导出、税费社保、绩效奖金、通知模板
- **原因**: 金额单位和快照规则一旦不一致，会造成跨系统对账差异。

## 禁区

| 路径或做法                              | 原因                                      | 替代动作                                      |
| --------------------------------------- | ----------------------------------------- | --------------------------------------------- |
| 在 React 组件中访问数据库或拼接查询     | UI 层会绕过权限、事务和业务规则           | 通过 API client 或 server action 调用业务入口 |
| 在 `route.ts` 中堆复杂业务流程          | HTTP 层难以复用和测试，也容易遗漏事务边界 | 下沉到对应模块的 service / use-case           |
| 跨模块直接写入薪酬、考勤或绩效表        | 会绕过模块内快照、锁定和审计规则          | 通过模块 service 或事件契约完成               |
| 使用 JavaScript 浮点数计算薪资金额      | 浮点误差会导致工资单和财务导出不一致      | 使用分为单位的 integer 或 decimal string      |
| 未确认权限就返回薪资、证件、绩效字段    | 人资数据属于高敏感数据                    | 先走 Auth 数据范围和字段级权限校验            |
| 修改 `node_modules/`, `.next/`, `dist/` | 第三方依赖或构建产物不应手工编辑          | 修改源代码、依赖版本或构建配置                |
