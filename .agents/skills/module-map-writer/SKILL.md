# Module Map Writer

使用本 skill 生成或维护 `docs/module-map.md`。模块地图是给 AI coding agent 和工程师看的系统导航入口：帮助它们快速判断业务变更可能涉及哪些模块、哪些边界不能破、下一步该读哪些文档或搜索哪些代码。

## 核心原则

模块地图是地图，不是百科。

- 写稳定的大边界，不写完整文件树。
- 写模块职责和跨模块关系，不写函数、类、字段、实现细节。
- 写业务域和关键支撑域，不穷举所有目录。
- 写 AI 容易走错的硬边界、禁区和隐性勾连。
- 用链接指向详细文档，不在地图里展开长解释。

语义搜索负责找到具体代码；模块地图负责提供方向感、责任归属和边界判断。

## 工作流程

1. 明确目标。
   默认创建或更新 `docs/module-map.md`。若用户指定其他路径，按用户路径处理。默认使用中文正文，保留必要英文术语。

2. 收集高价值上下文。
   先读取现有入口和知识文档，优先级如下：
   - 根 `AGENTS.md`、`ARCHITECTURE.md`、已有 `docs/module-map.md`
   - `docs/architecture/`、`docs/adr/index.md`、影响地图、业务规约目录
   - 项目顶层目录和主要业务目录
   - 必要时使用 `rg`、语义搜索或文件搜索确认模块位置

   不要为了写地图而深挖所有代码。只做足以识别大模块、边界、禁区和指针的调查。

3. 选择粒度。
   只列 vital few：
   - 核心业务域
   - 关键支撑域
   - 横切基础设施
   - 禁区或废弃区
   - 明确影响多个模块的硬边界

   若某模块内部复杂，顶层地图只给一行和指针，例如 `docs/modules/order.md`。不要在顶层展开子目录。

4. 编写地图。
   起草前读取 `references/module-map-template.md`。保持短、硬、可导航。每个模块条目通常只写路径、职责、指针和必要的隐性勾连。

5. 处理依赖和边界。
   模块地图可以写跨模块/跨领域硬边界，例如 “Order 不直接修改 Billing 状态”。不要把普通垂直分层规范写成大段编码规则。若有详细依赖规范，链接到 `docs/architecture/dependency-rules.md`。

6. 自检。
   交付前读取 `references/module-map-checklist.md` 并自查。重点确认地图没有变成文件树、README、ADR、编码规范或长篇模块说明。

## 内容去向判断

当某个内容不确定应放在哪里时，按下表判断：

| 内容                                  | 应放位置                                  |
| ------------------------------------- | ----------------------------------------- |
| 当前必须遵守的一句话规则              | `AGENTS.md`、`ARCHITECTURE.md` 或规范文档 |
| 模块职责、路径、入口导航              | `docs/module-map.md`                      |
| 跨模块硬边界、AI 常走错的禁区         | `docs/module-map.md` 或架构文档           |
| “改 A 可能暗伤 B”的隐性跨模块提醒     | `docs/impact-map.md`                      |
| 架构决策背景和取舍                    | ADR                                       |
| 业务行为、场景、验收规则              | 业务规约                                  |
| 分层 import、依赖方向、可自动检查规则 | 依赖规则文档、linter 或 sensor            |
| 完整测试列表、发布前必跑项            | 测试文档、CI、PR checklist                |
| 函数、类、字段、实现细节              | 代码、模块子文档或注释                    |

## 质量标准

一份合格的 `docs/module-map.md` 应满足：

- agent 读完后知道从哪些模块开始调查。
- agent 知道哪些跨模块方向不能走。
- agent 知道哪些目录不该编辑或模仿。
- agent 知道是否需要继续读取影响地图。
- 每一行都能解释“没有这一行，agent 可能会犯什么错”。
- 顶层地图短而稳定；完整性靠子文档、语义搜索和后续 ratchet 演化补足。
