# QA 验收指引写作

## 概览

创建一份 Markdown 指引，帮助 QA、UAT、产品验收和实施验收人员理解“应该验证什么”，而不需要反复解读业务语言很重的源文档。把 epic 和 phase guide 中的规则转成可观察的规则、步骤行为、异常路径和验收证据。

## 核心标准

为“需要判断功能是否可以验收通过的人”写，而不是为设计者或开发者写。

每条重要规则至少回答下面三个问题：

- **什么时候**适用这条规则？
- **谁或什么动作**会触发它？
- **用户应该看到什么，或系统应该做什么？**
- **哪些数据、状态、权限或记录应该变化？**
- **什么事情不得发生？**
- **QA 如何证明它通过了？**

## 源文档阅读流程

1. 定位并阅读权威源文档，通常包括 epic 史诗文档、phase implementation guide 阶段实施指引、proposal/spec，以及源文档中链接的规则文档。
2. 提取功能范围、用户角色、操作入口、状态流转、数据规则、权限、外部依赖、异常路径、上线或迁移注意事项。
3. 区分“源文档已确认的规则”和“仍需确认的假设”。不要编造源文档没有说明的业务行为、测试数据、接口名、UI 文案或数据库字段。
4. 把业务语言改写成 QA 语言：前置条件、操作动作、期望结果、可见证据和风险。
5. 保留源文档中会影响验收的精确值，包括状态、枚举、阈值、时间窗口、字段名、品牌和错误条件。
6. 把未解决或有歧义的内容放入简短的“待确认事项”，不要用顺滑的文字掩盖不确定性。

## 文档结构

撰写完整指引时读取 `references/template.md`。需要参考语气和改写力度时读取 `references/example.md`。

推荐章节：

1. 文档目的与适用范围
2. 测试对象与不覆盖范围
3. 关键术语速查
4. 角色、入口与前置条件
5. 流程规则总览
6. 分步骤验收规则
7. 状态、权限与数据规则
8. 异常场景与边界条件
9. 回归关注点
10. 验收清单
11. 待确认事项

## 版式选择

默认使用常规 Markdown 写法：标题、短段落和列表。表格不是默认格式，只在它能明显降低理解成本时使用。

优先使用常规 Markdown 的场景：

- 一个检查点需要解释背景、原因、连续步骤或多个注意事项。
- 单元格内容会超过两句短句，或需要换行才能读懂。
- 内容更适合按“前置条件 -> 操作 -> 期望结果 -> 注意事项”展开。
- 读者需要顺序阅读，而不是横向比较。

适合使用表格的场景：

- 状态流转、权限矩阵、字段规则、异常场景清单。
- 多条规则共享同一组维度，例如“场景/触发方式/期望结果”。
- 每个单元格都能保持短、准、可扫描。

## 写作规则

- 不要把整份指引写成表格集合。先用自然的 Markdown 层级组织内容，再为确实需要横向比较的规则使用表格。
- 使用面向测试人员的动词：`检查`、`确认`、`触发`、`观察`、`验证`、`不得`、`应展示`、`应记录`。
- 可以保留业务术语，但第一次出现时必须立刻翻译成可测试含义。
- 明确写出反向规则：禁止流转、禁止编辑、防重复、无变化行为、幂等预期和回滚限制。
- 写清验收证据：页面结果、接口响应、数据库/日志记录、通知、导出文件、下游同步状态，或确认没有发生变化。
- 保持“知道实现存在”，但不要写成开发文档。只有当字段、状态、任务、路由、日志或系统能帮助 QA 验证行为时才提及。
- 避免营销语言、设计动机、架构叙事和过长业务背景。
- 除非用户明确要求，不要把指引写成完整测试用例集。这份文档应帮助测试人员设计和执行用例，不应编造详细 case 编号或测试数据集。

## 提取清单

写作前，从源文档中扫描：

- 操作人和权限
- 页面、接口、命令、任务、导入/导出或同步入口
- 必填字段和选填字段
- 状态与状态流转
- 品牌、租户、区域、客户、企业或账号边界
- 创建、更新、删除、审批、取消、重试和失败规则
- 去重、匹配、覆盖和合并行为
- 日志、通知、下游推送、异步任务、审计记录等副作用
- 历史数据、迁移、兼容性和回滚注意事项
- 明确不做的内容和阶段边界

## 质量门槛

最终指引必须满足：

- QA 能看出主流程、关键分支和非法路径分别要测什么。
- 每个主要步骤都有清晰的前置条件、操作动作、期望结果和验收证据。
- 源文档中的业务术语被简化，但含义没有被改写。
- 歧义被集中放进明确问题中。
- 文档能区分当前阶段范围、未来阶段范围和不覆盖范围。
- 版式自然可读，长内容不要塞进表格；表格只用于短规则、矩阵和清单。
- 文档以 `.md` 保存到用户要求的文件夹；当用户说“之前的文件夹”或“同一个文件夹”时，保存到 epic/phase guide 所在文件夹。

## 文件命名

必须使用： `qa-acceptance-guide.md`

代码仓库中如果周边文件名以英文为主，优先使用英文文件名。团队日常中文沟通时，文档标题优先使用中文。

## 自检

完成前检查：

- epic 或 phase guide 中的重要规则没有遗漏。
- 没有添加源文档不支持的细节。
- 结构可读，不因表格过多或过宽影响阅读。
- 文档包含足够的反向规则和边界场景。
- 标题和文件名能让人一眼看出这是面向 QA 和验收人员的文档。
