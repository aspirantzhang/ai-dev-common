# QA/验收指引写作样例

用这个样例对齐语气和颗粒度。样例刻意保持通用；实际写作时应按源文档替换术语和规则。

## 源规则

> 企业在被认证后会转为客户。认证过程中需要保留企业来源信息，并根据品牌判断内外贸归属。后续阶段再处理跨品牌合并。

## 面向测试/验收人员的改写

### 认证转客户规则

认证前，待认证对象必须仍是企业，不能已经是客户。验收时先在企业详情页或接口中确认对象处于未认证或待认证状态。对已认证客户再次发起认证时，系统应拦截，或保持原客户关系不变，不得重复创建客户。

认证通过后，系统应创建或关联客户记录。验收人员需要确认：

- 客户列表中能查到对应客户。
- 客户核心字段与认证前企业信息一致。
- 企业来源信息没有丢失。
- 认证动作没有误更新其他品牌或其他企业的数据。

品牌归属按源文档中的品牌规则验证：

- 品牌 1、4 按内贸验证。
- 品牌 2、3 按外贸验证。
- 不要自行扩展源文档没有说明的品牌分组。

跨品牌合并不在本阶段验收。本阶段只确认系统不会误合并跨品牌数据；跨品牌合并规则应放入待确认事项或后续阶段。

### 异常与边界

异常场景较短且维度统一，可以使用表格：

| 场景 | 触发方式 | 期望结果 |
| --- | --- | --- |
| 重复认证 | 对已认证对象再次发起认证 | 系统应拦截或保持原客户关系不变 |
| 缺少品牌 | 使用无品牌或品牌异常的数据认证 | 系统应提示错误或阻止进入下一步 |
| 跨品牌相似企业 | 准备两个名称相近但品牌不同的企业 | 本阶段不得自动合并 |

### 待确认事项

- 重复认证的具体提示文案是否已有统一标准。这个问题会影响 UI 验收和自动化断言，建议向产品或开发确认。
- 认证失败后是否需要写入日志。这个问题会影响异常场景的验收证据，建议向开发确认。
