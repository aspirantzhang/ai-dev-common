# 阶段实施指引

## 目标

编写可交给后续开发者逐阶段执行的阶段实施指引。它是史诗文档的配套文档：史诗文档说明长期目标和阶段计划，阶段实施指引说明每个阶段开始前应遵守、调查、保护、复核和验收什么。

核心原则：把已确认事实、业务决策、代码参考、推断风险和待确认问题分开写。不要把未经验证的判断包装成结论。

## 工作流程

1. 先确认该文档服务于哪个 epic、哪些阶段、哪些业务范围。
2. 收集用户已明确拍板的业务规则，把它们写入“已确认业务规则”。
3. 对需要技术落地的阶段，先做必要代码调查；未调查充分时，只写“代码调查入口”或“需复核”，不要写成必须修改的文件。
4. 每个阶段使用固定六段结构。
5. 写完后检查措辞：确定事项要明确，未确定事项要显式标注，验收关注要偏结果而不是偏实现。

## 每阶段固定结构

每个阶段优先使用固定六段结构，但六段只是分类容器，不要求每段都写满；简单阶段应保持简短，某段没有真实内容时不要硬写。
复杂阶段可以写得充分；简单阶段可以每段 0-2 条。没有明确价值的“不得破坏”“排查点”“风险点”“验收点”不要编造。

### 已确认业务规则（必须遵守）

写用户已明确确认、代码已证实或规约已确定的业务规则。这里的内容是后续开发必须遵守的硬规则。

### 代码调查参考（代码调查入口）

写后续 AI 开始调查时应优先查看的文件、方法、命令、Resource、Action、Job、脚本或测试。措辞使用“入口”“重点检查”“参考”，不要暗示只改这些文件就够。

### 不得破坏的行为（必须保持不变的行为）

写阶段改造时必须保护的既有行为，例如兼容旧字段、默认值、删除逻辑、日志、灰度、幂等、事务、权限、异常处理、已有调用方契约。
只写与本阶段改造直接相关、且确实可能被误伤的既有行为；不要把通用稳定性要求泛泛列入每个阶段。

### 阶段开始前建议排查的点

写开工前应先确认的调查动作。适合放“全局搜索”“确认实际运行范围”“确认表字段是否存在”“检查测试覆盖”等。

### 需复核或确认的业务行为或风险点

写未被充分证实、可能影响方案的点。这里必须诚实表达不确定性，例如“需确认”“当前代码参考未明显看到”“如果实际触达则需另行确认”。

### 验收方向建议（验收关注）

写完成后应证明的业务结果，不写过细实现步骤。验收关注应覆盖成功路径、兼容路径、异常路径和不可破坏行为。
验收关注优先覆盖本阶段核心业务结果；只有当兼容、异常或不可破坏行为与本阶段直接相关时才补充。

## 应该写什么

- 写已确认的业务口径和字段映射。
- 写后续开发的代码调查起点。
- 写现有系统中不能被改坏的行为。
- 写开工前必须复核的入口、字段、运行范围和测试。
- 写风险和不确定项，并明确它们不是结论。
- 写结果型验收方向，帮助后续开发证明没有跑偏。

## 不应该写什么

- 不要把未经调查的推断写成已确认规则。
- 不要写“只需要改某文件”这类容易误导的绝对判断，除非已经完整排查。
- 不要把当前代码现状写成目标行为。
- 不要在未拍板时写死算法、字段转换或外部系统行为。
- 不要写与该阶段无关的实现细节、代码风格、长篇背景或重复史诗文档内容。
- 不要把验收关注写成具体代码改法。
- 不要为了维持模板完整而硬凑内容；没有实际指导价值的通用提醒，应省略或写成“无特别补充”。

## 写作口径

- 用“已确认”“需复核”“当前代码参考”“重点检查”等词区分确信度。
- 已确认规则要短、明确、可执行。
- 代码调查参考要给入口，不假装穷尽所有文件。
- 风险点要说明为什么风险存在，以及后续开发前应如何确认。
- 验收方向要描述业务结果，例如“不丢地址 2 至地址 5”“异常时不创建半成品”“旧调用方仍可读取原字段”。

## 阶段模板

```markdown
## Phase N: 阶段名称

已确认业务规则（必须遵守）：

- ...

代码调查参考（代码调查入口）：

- ...

不得破坏的行为（必须保持不变的行为）：

- ...

阶段开始前建议排查的点：

- ...

需复核或确认的业务行为或风险点：

- ...

验收方向建议（验收关注）：

- ...
```

## 与史诗文档的关系

不要在阶段实施指引中重复史诗文档的完整背景、阶段进度或实现状态。史诗文档记录长期目标、阶段计划和进度；阶段实施指引用于沉淀每个阶段的执行护栏和调查入口。

如果用户要求同时维护 epic 和阶段实施指引：先保证 epic 的阶段、状态和进度准确，再把能指导实施的细节写入阶段实施指引。
