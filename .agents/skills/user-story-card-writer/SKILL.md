# User Story Card Writer

## Purpose

把业务化、口语化的需求整理成可读的用户故事卡，并尽量贴近项目里已有的故事卡结构。

## Workflow

1. 先读 `.develop/AGENTS.md`，确认项目规约、目录和禁区。
2. 先查规约，再想实现。
   - 只看 `.develop/specs/` 目录下的规约。
3. 先找同类能力和已有写法，再决定这次卡片怎么写。
   - 如果用户给了参考文档，优先用飞书工具读取它
   - 如果文档在飞书里，使用 `lark-cli docs +fetch --api-version v2`
   - 不要用公开网页搜索替代私密飞书文档
4. 把需求翻译成用户能直接读懂的故事卡，不写实现方案。
5. 默认输出结构保持简单清楚：
   - `# 背景`
   - `# 用户故事`
   - `# 验收标准`
   - `# Given-When-Then 验收场景`
   - `# 待确认`
6. 验收标准要写成业务结果，不要写代码步骤。
7. 场景要尽量具体，最好能直接拿去做测试或评审。
8. 不确定的地方保留在 `## 待确认`，不要硬编结论。
9. 如果用户要放进飞书，再把同一份内容创建或导入到目标文件夹。

## Feishu Docs

- 用户给飞书链接或 token 时，使用飞书工具读取，不要用公开网页搜索。
- 读取私密飞书文档：`lark-cli docs +fetch --api-version v2 ...`
- 查找目标文件夹、同类文档或已有卡片：`lark-cli drive +search ...`
- 在指定文件夹创建在线文档：`lark-cli docs +create --api-version v2 --parent-token <folder_token> ...`
- 修改已有在线文档：`lark-cli docs +update --api-version v2 ...`
- 如果已先生成本地 Markdown/Word，再导入飞书：`lark-cli drive +import --type docx --parent-token <folder_token> ...`
- 找不到文件夹或没有权限时，说明卡住的位置和需要用户补充的信息，不要换成网页搜索。

## Writing Rules

- 语言要正常、直接，别堆术语。
- 不要把规约写成技术方案。
- 不要把猜测伪装成事实。
- 如果信息足够，就直接写；如果信息不够，就补 `# 待确认`。
- 例子只作为写法参考；真正结论必须来自当前用户需求和当前规约。

## 标题规则

- 文档标题必须先确定，并且只出现一次。
- 写入飞书文档时，标题由文档标题位承载；正文从“背景”开始，不再重复同名标题。
- Markdown 模式下，标题使用第一行 `# 用户故事卡：{主题}`。
- XML 模式下，标题使用 `<title>用户故事卡：{主题}</title>`。

## Good Output Shape

- 背景：一句到两句，说明为什么要做。
- 用户故事：用角色 + 目标 + 价值来写。
- 验收标准：按主题分组，写清楚“什么情况下，系统应该怎样”。
- 场景：用 Given / When / Then 展开关键路径。
- 待确认：列出仍然模糊的问题。

## Example

用户输入：

```text
修改内外勤销售及内贸客户区域，父级有变更时：
若下级客户内外勤及内贸客户区域与父级客户相同则下级客户相关字段同步变更
若下级客户内外勤及内贸客户区域与父级客户不同则下级客户相关字段不变更
```

可输出：

```markdown
# 用户故事卡：父级销售归属变更时同步同源下级客户

## 背景

父级客户与下级客户可能共享同一组内外勤销售和内贸客户区域。父级客户调整这些归属信息时，需要避免下级客户仍停留在旧的同源配置，同时也不能覆盖已经独立维护过的下级客户配置。

## 用户故事

作为客户管理人员，我希望父级客户的内外勤销售和内贸客户区域变更后，系统只同步仍与父级保持一致的下级客户，这样既能减少重复维护，也能保留下级客户的差异化管理结果。

## 验收标准

1. 当父级客户的内勤销售、外勤销售或内贸客户区域发生变更时，系统识别该父级客户下的下级客户。
2. 如果某个下级客户在变更前的相关字段与父级客户变更前的值相同，则该下级客户对应字段同步更新为父级客户变更后的值。
3. 如果某个下级客户在变更前的相关字段与父级客户变更前的值不同，则该下级客户对应字段保持不变。
4. 同步判断应以字段为单位，只有仍与父级保持一致的字段才跟随父级变更。

## Given-When-Then 验收场景

### 场景 1：下级客户字段与父级原值一致时同步

- Given 父级客户的内勤销售为 A
- And 下级客户的内勤销售也为 A
- When 用户将父级客户的内勤销售改为 B
- Then 下级客户的内勤销售同步改为 B

### 场景 2：下级客户字段与父级原值不一致时不同步

- Given 父级客户的外勤销售为 A
- And 下级客户的外勤销售为 C
- When 用户将父级客户的外勤销售改为 B
- Then 下级客户的外勤销售仍为 C

## 待确认

- “内外勤销售及内贸客户区域”具体包含哪些字段，需要以规约中字段定义为准。
- 当字段为空值时，是否也按“相同则同步”的规则处理。
```
