# Impact Map Writer

使用本 skill 生成或维护 `docs/impact-map.md`。

影响地图是给 AI coding agent 的轻量预警表，不是依赖图、模块地图、ADR、测试清单或风险台账。它只记录一类事情：**改 A 时，Agent 或新人不容易自然想到，但确实可能影响 B 的跨模块关系**。

## 核心原则

- 只写非显而易见的跨模块影响。
- 只写会造成真实后果的业务语义变化、异步事件、共享状态、数据契约或副作用。
- 不写普通 import、显式调用、同模块内部联动、类型系统或 linter 能稳定发现的问题。
- 不堆砌猜测；正式条目应来自历史问题、代码线索、issue、PR、规约、测试或人类确认。
- 每条只回答四件事：触发什么、影响谁、为什么、怎么检查或应对。

## 工作流程

1. 明确目标文件。
   默认创建或更新 `docs/impact-map.md`。如果用户指定其他路径，按用户路径处理。正文默认使用中文，保留必要英文术语。

2. 快速读取必要上下文。
   优先查看 `docs/module-map.md`、已有 `docs/impact-map.md`、相关规约、ADR 索引、issue、PR、incident/post-mortem、跨模块测试。只读取与本次任务有关的材料，不做全库审计。

3. 找隐性影响源。
   重点关注事件、消息队列、Webhook、定时任务、共享缓存、搜索索引、报表、对账、状态机、枚举、权限、金额精度、生命周期变化。

4. 判断是否写入。
   同时满足这些条件才写正式条目：跨模块、非显而易见、有实际后果、有具体应对动作、有线索或确认。否则不要写，必要时列为待确认问题。

5. 编写影响地图。
   创建或重写时读取 `references/impact-map-template.md`。默认使用分节式 Markdown 条目；每条用短标题和固定字段表达。条目超过约 15 条时，再按触发模块分组。不要默认引入编号、严重度、状态流转或归档区。

6. 交付前自检。
   需要检查时读取 `references/impact-map-checklist.md`。重点压缩空泛提醒和重复内容，保证每条都能指导 Agent 下一步检查。

## 内容去向判断

当某个内容不确定应放在哪里时，按下表判断：

| 内容                                  | 应放位置                                  |
| ------------------------------------- | ----------------------------------------- |
| 当前必须遵守的一句话规则              | `AGENTS.md`、`ARCHITECTURE.md` 或规范文档 |
| 模块职责、路径、入口导航              | `docs/module-map.md`                      |
| 跨模块硬边界、AI 常走错的禁区         | `docs/module-map.md` 或架构文档           |
| “改 A 可能暗伤 B”的隐性跨模块提醒     | `docs/impact-map.md`                      |
| 架构决策背景和取舍                    | ADR                                       |
| 业务行为、场景、验收规则              | 业务规约                                  |
| 分层 import、依赖方向、可自动检查规则 | 依赖规则文档、linter 或 sensor            |
| 完整测试列表、发布前必跑项            | 测试文档、CI、PR checklist                |
| 函数、类、字段、实现细节              | 代码、模块子文档或注释                    |
