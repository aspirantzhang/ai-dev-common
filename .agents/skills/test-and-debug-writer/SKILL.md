# 运行手册写作

## 定位

编写一份面向 AI Agent 的运行手册，让系统能进入“可运行、可观察、可验证”的状态。运行手册不是测试规范的替代品：

- 测试规范说明如何写测试、如何设计断言、如何判断测试输出。
- 运行手册说明如何启动系统、执行命令、观察状态、定位环境问题、选择验证路径。

运行手册属于 harness 的 guides 与 sensors 之间的连接层：它把开发环境、命令入口、日志、健康检查、seed/reset、常见失败处理这些隐性知识显式化，降低 agent 的盲试、假阳性通过、token 漂移和破坏性操作风险。

## 写作原则

把运行手册写成机器可执行 SOP，而不是散文说明。

- 优先指向脚本、Makefile target、npm script、容器命令；不要把可脚本化步骤写成长段文字。
- 先查已有入口，再写手册；不要自行发明启动、测试、迁移或清理命令。
- 每个命令都必须说明运行位置：宿主机、容器内、指定服务容器、CI、本地开发环境。
- 每个启动命令都必须配一个自检命令，例如健康检查 URL、进程状态、日志关键词、数据库连接检查。
- 每个重置、seed、migration、缓存清理命令都必须标注适用环境，默认只允许 dev/test。
- 破坏性 reset、清库、重建卷、删除缓存目录必须标注停止条件和人工确认点。
- 不写真实凭证、生产地址、生产数据库操作、不可审计的清理脚本。
- 保持短：根文档通常控制在 30-80 行，真正复杂的知识沉淀到脚本、target 或专门文档中。

## 工作流程

1. 先读现有入口：`AGENTS.md`、`package.json`、`docker-compose*.yml`、`Makefile`、`scripts/`、`.env.example`、`docs/` 中已有运行、测试、部署、MCP 或本地开发说明。
2. 识别现有边界：测试规范、模块地图、影响地图、ADR、领域语义文档已有的内容不要重复搬运，只在运行手册中链接。
3. 选择目标文件：优先更新项目已有运行手册；没有则创建 `docs/runbook.md` 或用户指定路径。
4. 把“步骤”压缩成“入口”：如果发现需要写 5 行以上操作步骤，优先建议新增脚本或 Makefile target，再在手册中引用。
5. 为每个关键入口补上观察点：启动后看什么 URL、日志、容器、队列、DB 表或健康检查才能证明系统真的可用。
6. 为常见失败补异常 playbook：只写高频、可判定、可修复的问题，不写百科式排障。
7. 补服务依赖顺序：数据库、Redis、队列、外部 API mock、后端、前端分别如何启动和自检。
8. 最后补验证矩阵：最小验证、完整验证、按变更类型追加验证。

- 同一个命令连续失败 2 次后，停止盲试，改查日志、环境变量、容器状态和 Playbook。

- 需要执行 reset、清库、删除卷、清缓存目录时，先确认环境是 dev/test，并说明影响范围。

## 内容边界

保留在运行手册中：

- 启动、停止、重启、自检。
- 容器命令格式、服务名、端口、健康检查。
- 日志查看、运行时状态查看、常见环境故障判断。
- 本地 seed/reset、dev/test migration、缓存清理。
- 变更后的最小验证和完整验证入口。

不要放入运行手册：

- 领域概念定义：放 `docs/domain.md`。
- 模块职责和边界：放 `docs/module-map.md`。
- 跨模块暗伤和同步测试提醒：放 `docs/impact-map.md`。
- 架构取舍原因：放 ADR。
- 测试设计规范：放测试 skill 或测试规范文档。
- 真实账号、Secret、生产连接串、生产清理步骤。

## 项目特化策略

对于容器化项目，优先把容器执行格式写进“命令约定”，例如：

```bash
docker exec <container> bash -c "su www-data -c '<your-command>'"
```

然后在后续表格里只写 `<your-command>` 或封装脚本，避免每一行重复长命令。若容器名、端口、服务名会变化，要求使用 `.env.example`、Makefile、脚本或 compose service name 作为唯一事实来源。

对于 AI 开发示例项目，运行手册要突出三件事：

- agent 应该从哪个入口启动和验证，而不是自己猜命令。
- agent 遇到环境错误时先查哪个 playbook，而不是盲目重装依赖或清空目录。
- agent 完成变更后如何证明系统真的跑过、看过、测过。
- 外部服务不可用时，agent 应该使用哪个 mock、fixture 或本地替身，而不是访问真实第三方系统。

## 质量检查

提交运行手册前检查：

- 是否每个启动入口都有自检。
- 是否每个命令都有运行位置和环境边界。
- 是否明确了服务依赖顺序和外部 API 的本地替身策略。
- 是否写明失败停止条件，避免 agent 进入盲试循环。
- 是否有任何真实凭证、生产地址或生产操作。
- 是否重复了其他知识库文档的职责。
- 是否把长步骤替换成脚本、Makefile target 或 npm script。
- 是否能让一个没有项目经验的 agent 在 5 分钟内找到启动、观察、排障、验证入口。
