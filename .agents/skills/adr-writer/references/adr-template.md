# ADR 精简模板

创建或更新 ADR 时使用本模板。默认写入 `docs/adr/NNN-short-decision-slug.md`。

```markdown
# ADR-NNN: <用一句话概括决策>

- 状态：Proposed | Accepted | Superseded by ADR-NNN | Deprecated
- 日期：YYYY-MM-DD
- 适用标签：<database, billing, dependency, deployment 等>

## 背景

说明当时遇到的具体问题、系统压力、约束条件，以及为什么现在必须做这个决策。
控制在 3 到 6 句话，不写技术教程。

## 决策

明确写出最终选择。

主要理由：
- <理由 1>
- <理由 2>

被放弃的替代方案：
- <替代方案>：放弃原因是 <原因>
- <替代方案>：放弃原因是 <原因>

## 影响

正面影响：
- <收益>
- <收益>

已接受的代价：
- <成本、限制或技术债>
- <成本、限制或技术债>

影响范围：
- <涉及的模块、边界、数据、部署或流程>

## 后续

重新评估触发条件：
- <具体、可观察的条件>

发现入口和约束：
- <index、AGENTS.md、模块地图、架构文档、lint、测试或 review 指针>
```

## 质量标准

- 短而清楚，通常控制在一页以内。
- 不写官方文档摘要，不写实现教程，不复制长篇讨论过程。
- 存在真实替代方案时，必须写出至少一个替代方案和放弃原因。
- “已接受的代价”不能空；没有代价通常说明 ADR 不值得写或还没想清楚。
- 重新评估触发条件要具体，不能只写“未来需要时”。
- 热路径文档只放短结论和链接，不复制 ADR 的推理过程。
